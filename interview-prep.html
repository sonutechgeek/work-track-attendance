Node.js & Express 10 questions
BasicWhat is middleware in Express.js?⌄

Middleware is a function that runs between the request arriving and the response being sent. It has access to req, res, and next. You call next() to pass control to the next middleware, or next(err) to jump to the error handler.

const myMiddleware = (req, res, next) => { // do something with req/res next() // pass to next middleware }

In this project: helmet → cors → rateLimiter → authenticate → authorize → validate → controller — that's the middleware chain for a protected route.

BasicWhat is the difference between app.js and server.js in your project?⌄

app.js — creates the Express app, registers all middleware and routes. It's a pure configuration file with no network I/O. It just exports the app function.

server.js — creates the HTTP server, attaches Socket.io, binds the port, connects to the database, and handles graceful shutdown.

Why separate them? You can import app.js alone in tests without actually starting a server and binding a port. Socket.io also needs the raw http.Server object, not the Express app.
BasicWhat does next(err) do vs next()?⌄

next() — passes control to the next middleware in the chain (normal flow).

next(err) — skips all normal middlewares and jumps directly to the error handler (a middleware with 4 arguments: err, req, res, next). In this project that's errorHandler.js.

if (!user) return next(new AppError('User not found', 401)) // ↑ Jumps straight to errorHandler, controller never runs
BasicWhat is CORS? How did you configure it?⌄

CORS (Cross-Origin Resource Sharing) is a browser security rule that blocks a website from calling an API on a different domain unless the server explicitly allows it.

In this project: only http://localhost:5173 (the React frontend) is allowed. Only GET, POST, PATCH, DELETE methods are permitted.

app.use(cors({ origin: process.env.CORS_ORIGIN, credentials: true, // allow cookies to be sent methods: ['GET', 'POST', 'PATCH', 'DELETE'], }))
MediumWhat is rate limiting? What limits did you set in this project?⌄

Rate limiting restricts how many requests a client can make in a time window. It protects against brute-force attacks, DoS attacks, and API abuse.

In this project there are two limiters:

• loginLimiter — 5 login attempts per 15 minutes (stops brute-force on passwords)
• apiLimiter — 100 requests per 15 minutes on all /api routes

Interview trick: Both limiters are disabled in development mode (NODE_ENV === 'development') so they don't block you while coding.
MediumWhat is graceful shutdown? How did you implement it?⌄

Graceful shutdown means stopping the server cleanly — finishing in-flight requests and closing DB connections — instead of killing it instantly.

SIGTERM — sent by Docker/PM2/Kubernetes when they want to stop the process.
SIGINT — sent when you press Ctrl+C in terminal.

process.on('SIGTERM', () => shutdown('SIGTERM')) const shutdown = async (signal) => { server.close(async () => { await prisma.$disconnect() // close DB cleanly process.exit(0) }) }
MediumWhat is process.on('unhandledRejection') and why do you need it?⌄

When a Promise rejects and nobody catches it with .catch() or try/catch, it becomes an unhandled rejection. Without this handler, Node.js would silently continue running in a broken state.

In this project it logs the error and triggers graceful shutdown — so the process exits cleanly instead of running with unknown corrupted state.

MediumWhat does express.json({ limit: '10kb' }) do?⌄

It parses incoming JSON request bodies and puts the result in req.body. The limit: '10kb' rejects any request body larger than 10 kilobytes.

Why the limit? Without it, an attacker could send a 100MB JSON body to crash the server or exhaust memory — called a body bomb / payload attack.

Deep DiveWhat is Helmet.js and what headers does it set?⌄

Helmet sets 14 HTTP security response headers automatically. Key ones:

• X-Content-Type-Options: nosniff — prevents MIME sniffing attacks
• X-Frame-Options: DENY — prevents your site from being iframed (clickjacking)
• Strict-Transport-Security — forces HTTPS (HSTS)
• Content-Security-Policy — restricts what scripts can run (XSS protection)
• X-XSS-Protection — legacy XSS filter for older browsers

Just say: "Helmet adds security headers that protect against XSS, clickjacking, and MIME sniffing. One line of code covers 14 headers."
Deep DiveWhat is Winston logger? What is DailyRotateFile?⌄

Winston is a structured logging library. Unlike console.log, it supports log levels (error, warn, info, debug), multiple outputs (console + file), and JSON formatting.

DailyRotateFile creates a new log file every day automatically. Old files are deleted after the retention period. In this project: error logs kept 30 days, combined logs kept 14 days.

In production, the log level is set to warn — so debug/info messages are suppressed to keep log files small.

JWT & Authentication 10 questions
BasicWhat is JWT? What are its three parts?⌄

JWT (JSON Web Token) is a compact, signed token used to verify identity without hitting the database on every request.

Three parts separated by dots: header.payload.signature

• Header — algorithm used (e.g. HS256)
• Payload — data inside (userId, role, expiry)
• Signature — HMAC of header+payload using the secret key. Verifies nobody tampered with the token.

The payload is only Base64 encoded — NOT encrypted. Anyone can decode it. Never put passwords or sensitive data in JWT payload.
BasicWhat is the difference between access token and refresh token?⌄

Access Token — short-lived (15 min). Sent in every API request header. Used to verify identity. Short expiry limits damage if stolen.

Refresh Token — long-lived (7 days). Stored in HttpOnly cookie. Used only to get a new access token when it expires. Never sent to API endpoints.

Flow: access token expires → frontend calls /auth/refresh with cookie → gets new access token → continues normally.

BasicWhat is an HttpOnly cookie? Why is the refresh token stored in one?⌄

An HttpOnly cookie cannot be read by JavaScript — only the browser sends it automatically with requests. This means even if your site has an XSS vulnerability (malicious script runs), it cannot steal the refresh token.

If the refresh token was stored in localStorage instead, any XSS attack could read it with localStorage.getItem('refreshToken').

MediumWhy do you bcrypt.hash the refresh token before saving to the database?⌄

If the database is compromised (SQL injection, backup leak), the attacker gets the stored refresh token. If it's stored as plaintext, they can use it to generate new access tokens and impersonate any user.

By storing only the bcrypt hash, even if the DB leaks, the attacker cannot reverse the hash to get the original token. On /refresh, we use bcrypt.compare(cookieToken, storedHash) to verify.

MediumWhat is account lockout? How did you implement it?⌄

Account lockout temporarily blocks a user after too many wrong password attempts — stops brute-force attacks.

Implementation: The users table has failedLogins (count) and lockUntil (timestamp) columns.

• Wrong password → increment failedLogins
• At 5 failures → set lockUntil = now + 30 minutes, reset counter
• On login → check if lockUntil > now, return 423 with minutes remaining
• Successful login → reset both fields to 0/null

MediumWhat is refresh token rotation?⌄

Every time the refresh token is used to get a new access token, the server issues a brand new refresh token and invalidates the old one. The new one is stored in the DB (as a hash) and set in the cookie.

Why? If an attacker steals the refresh token, they can use it once — but after that the legitimate user's next refresh call will have already rotated it, and the attacker's stolen token is now invalid.

MediumWhat happens when jwt.verify() fails? What errors can it throw?⌄

It throws one of two specific errors:

• TokenExpiredError — token signature is valid but the exp time has passed → return 401 "Access token expired"
• JsonWebTokenError — token is malformed, tampered with, or signed with wrong secret → return 401 "Invalid access token"

In the auth middleware, both are caught separately to return precise error messages.

MediumWhat does the authenticate middleware do step by step?⌄

1. Extract token from Authorization: Bearer <token> header
2. Call jwt.verify(token, JWT_SECRET) — verify signature + expiry
3. Fetch user from DB using decoded.id — confirm user still exists
4. Check user.isActive — reject deactivated accounts even with valid token
5. Attach req.user = user for controllers to use
6. Call next()

Why fetch from DB if JWT is already verified? Because the token might be valid but the user could have been deactivated after it was issued. JWT has no way to know that.
Deep DiveWhat is the difference between authentication and authorization?⌄

Authentication — "Who are you?" — verifying identity. In this project: JWT token verification in authenticate middleware.

Authorization — "What are you allowed to do?" — verifying permissions. In this project: authorize('ADMIN') role check + checkManagerScope team check.

Authentication always runs first. Authorization runs after — it needs to know who you are before deciding what you can do.

Deep DiveHow do you change password securely? Why verify the old password first?⌄

The PATCH /auth/me/password endpoint requires both currentPassword and newPassword.

Why verify old password? The user is already authenticated (has a valid JWT), but their session might have been stolen. Requiring the old password confirms the actual person is making this change — not someone who stole the session token. This prevents account takeover via session hijacking.

New password is hashed with bcrypt.hash(newPassword, 12) before saving. Cost factor 12 means it's slow enough to resist brute-force.

Socket.io 7 questions
BasicWhat is the difference between HTTP and WebSocket?⌄

HTTP — Request/Response. Client sends request → server replies → connection closes. Client must ask again for new data.

WebSocket — Persistent bidirectional connection. Client connects once → both sides can send data anytime → connection stays open until one side closes it.

In this project: HTTP is used for all API calls (login, check-in). WebSocket (via Socket.io) is used for real-time events (timer start, leave notification).

BasicWhy did you use Socket.io in this project?⌄

Two reasons:

1. Live attendance timer — when an employee checks in, the timer must start instantly on their screen without a page refresh. With HTTP polling you'd need to call the server every few seconds — wastes resources and is laggy.

2. Real-time notifications — manager gets instant notification when an employee applies for leave. Admin dashboard updates live.

Always explain the business reason, not just the technology. "The requirement said the timer should continue accurately even after page refreshes — that's why Socket.io is used."
MediumWhat is a Socket.io room? How did you use them?⌄

A room is a named channel that sockets can join. You can emit an event to everyone in a room with one call.

In this project there are two types of rooms:

• user:{id} — each employee has a private room (e.g. "user:5"). Only their events go there. Employee 5 never receives employee 7's events.
• admin — all connected admins join this room. When any leave is applied, all admins get notified at once.

socket.join(`user:${socket.userId}`) if (role === 'ADMIN') socket.join('admin')
MediumHow do you authenticate Socket.io connections?⌄

Socket.io has its own middleware (separate from Express middleware). It runs before any connection is accepted.

The frontend sends the JWT access token in the handshake:

io('http://localhost:5000', { auth: { token: accessToken } })

The server middleware calls jwt.verify(token, JWT_SECRET). If valid, it attaches socket.userId and socket.userRole. If invalid, the connection is rejected before it's established.

MediumWhy does Socket.io need the raw http.Server and not the Express app?⌄

WebSocket connections start as HTTP but then send an Upgrade request to switch protocols. This upgrade is handled at the TCP/socket layer — below Express.

The http.Server fires an 'upgrade' event for this. Socket.io listens to that event. The Express app never sees upgrade requests — they're intercepted before reaching Express's request handler.

http.Server → fires 'upgrade' event → Socket.io takes over http.Server → normal HTTP request → Express handles it
Deep DiveHow does the live timer survive a page refresh?⌄

The timer state is in the database, not in memory. When the page loads, the frontend calls GET /attendance/live-timer which returns:

{ active: true, checkInTime: "2024-01-15T09:00:00Z", elapsedMs: 14400000 // Date.now() - checkInTime }

The frontend reads elapsedMs and resumes the timer from that exact offset. The source of truth is always the DB — not the browser. Refresh doesn't lose anything.

Deep DiveWhat is the getIO() function and why do you need it?⌄

getIO() returns the single shared Socket.io instance. The io variable is in module scope — initialized once when the server starts.

Why? Controllers (like attendance.controller.js) need to emit Socket events. They import emitToUser from socket.js. If io isn't initialized yet and something calls getIO(), it throws "Socket.io not initialized" — a clear error instead of a silent undefined.

Database & Prisma 10 questions
BasicWhat is Prisma ORM? Why use it instead of writing raw SQL?⌄

Prisma is an ORM (Object Relational Mapper) — it lets you query the database using JavaScript objects instead of writing SQL strings.

Benefits: type-safe queries (TypeScript catches errors), auto-generated client from schema, migrations management, prevents SQL injection by default (all values are parameterized).

// Prisma — safe, readable prisma.user.findUnique({ where: { email } }) // Raw SQL — error-prone, injection risk if concatenated "SELECT * FROM users WHERE email = '" + email + "'"
BasicWhat is a composite UNIQUE constraint? Where did you use it?⌄

A composite unique constraint means the combination of two columns must be unique — neither column alone needs to be unique.

Used twice in this project:

• attendances — UNIQUE(employeeId, date): one employee can only have one attendance record per day. Employee 5 can appear on Jan 1, and Employee 7 can also appear on Jan 1, but Employee 5 cannot appear twice on Jan 1.
• leave_balances — UNIQUE(employeeId, year): one balance row per employee per year.

BasicWhat is onDelete: Cascade vs onDelete: SetNull?⌄

onDelete: Cascade — when the parent is deleted, all related children are also deleted automatically. Used on attendances and leaves — if an employee is deleted, their records are deleted too.

onDelete: SetNull — when the parent is deleted, the FK column in the child becomes NULL instead of deleting the child. Used on departmentId and managerId — if a department or manager is deleted, employees are preserved but their department/manager becomes null.

SetNull = preserve data. Cascade = delete everything. Choose based on whether the child record has meaning without the parent.
MediumWhat is prisma.$transaction? Why did you use it in leave approval?⌄

A transaction groups multiple DB operations so they either all succeed or all fail together (atomicity). If any operation fails, all previous operations in the transaction are rolled back.

In leave approval, 3 things happen:

1. Update leave status → APPROVED
2. Create attendance rows for each leave day
3. Deduct leave balance

Without a transaction: if step 3 fails, the leave is marked approved and attendance rows exist, but the balance was never deducted. That's corrupted data. With $transaction, all 3 happen or none happen.

MediumWhat is upsert? Where did you use it and why?⌄

Upsert = insert if not exists, update if exists. One operation instead of checking first then inserting/updating.

prisma.attendance.upsert({ where: { employeeId_date: { employeeId, date } }, update: { checkInTime: new Date(), status: 'PRESENT' }, create: { employeeId, date, checkInTime: new Date() } })

Used in check-in (create today's record or update it if one already exists) and in leave approval (create attendance row for each leave day, or update if the employee already marked something).

MediumWhat is Promise.all? Why did you use it in the dashboard and pagination?⌄

Promise.all runs multiple async operations in parallel and waits for all of them to complete. Without it, each DB query runs one after another (sequential).

// Sequential — slow (sum of all query times) const records = await prisma.attendance.findMany(...) const total = await prisma.attendance.count(...) // Parallel — fast (time of slowest single query) const [records, total] = await Promise.all([ prisma.attendance.findMany(...), prisma.attendance.count(...) ])

Dashboard uses 6 queries in parallel. Without Promise.all, it would be 6× slower.

MediumWhat is a self-referential relationship? Where is it in your schema?⌄

A self-referential (self-join) relationship is when a table has a foreign key pointing to itself.

In the users table, managerId points to users.id — meaning a user can be the manager of other users. One manager can have many employees, all stored in the same table.

// User ID=5 is manager of User ID=8 { id: 5, name: 'Manager A', managerId: null } { id: 8, name: 'Employee B', managerId: 5 }
MediumWhat are ENUMs in the database? Which ones do you have?⌄

An ENUM is a column that only accepts a predefined set of string values. The DB rejects any value not in the list — data integrity at the DB level.

In this project:

• Role — ADMIN, MANAGER, EMPLOYEE
• AttendanceStatus — PRESENT, ABSENT, HALF_DAY, LEAVE, WFH, FIELD_VISIT
• LeaveType — CASUAL, SICK, HALF_DAY, EARLY_LEAVE, WFH, FIELD_VISIT
• LeaveStatus — PENDING, APPROVED, REJECTED

Deep DiveWhat is @@index in Prisma schema? Why did you add indexes?⌄

A database index speeds up queries on that column by creating a separate sorted data structure. Without an index, every query scans the entire table (full table scan). With an index, the DB jumps directly to matching rows.

In users: @@index([role]), @@index([departmentId]), @@index([managerId]) — these columns are frequently used in WHERE clauses (e.g. "find all employees of this manager").

Trade-off: indexes speed up reads but slow down writes (insert/update must update the index too). Index columns you filter by frequently.
Deep DiveWhat does groupBy do in Prisma? Where did you use it?⌄

groupBy is equivalent to SQL's GROUP BY — it aggregates rows that share a column value and lets you count/sum them.

Used in the admin dashboard:

prisma.attendance.groupBy({ by: ['status'], _count: { status: true } }) // Returns: [{status:'PRESENT', _count:{status:142}}, ...]

This gives you "how many PRESENT, how many ABSENT, how many LEAVE" for the whole month in one query — instead of separate count queries per status.

RBAC & Security 8 questions
BasicWhat is RBAC? How does it work in your project?⌄

RBAC (Role-Based Access Control) means permissions are assigned to roles, not individual users. Every user has a role, and that role determines what they can do.

In this project: Admin can do everything. Manager can only manage their own team. Employee can only access their own data. One middleware function enforces this:

router.get('/', authenticate, authorize('ADMIN'), getAllUsers) // Only ADMIN role can reach getAllUsers controller
MediumWhat is manager scope guard? What problem does it solve?⌄

The role guard only checks if you're a MANAGER. It doesn't check which employees you can access. Without scope guard, Manager A could view Manager B's team data by guessing employee IDs.

Scope guard verifies: "does this employee's managerId match the requesting manager's id?"

const target = await prisma.user.findFirst({ where: { id: targetId, managerId: req.user.id } }) if (!target) → 403 'Not your team member'

This is called horizontal privilege escalation prevention — same role, different data scope.

MediumWhat is input validation? What does stripUnknown do in Joi?⌄

Input validation ensures request body fields are the correct type, format, and range before they reach the controller. Done with Joi schemas.

stripUnknown: true silently removes any fields from the body that are NOT defined in the Joi schema. This prevents mass-assignment attacks:

// Attacker sends: { email, password, role: 'ADMIN' } // Schema only defines email + password // After stripUnknown: { email, password } — role removed ✓
MediumWhat is XSS? How does this project protect against it?⌄

XSS (Cross-Site Scripting) — attacker injects malicious JavaScript into your page. If the script runs, it can steal tokens from localStorage, read cookies, etc.

Protection in this project:

• HttpOnly cookies — refresh token can't be read by JS, even if XSS runs
• Helmet's Content-Security-Policy — restricts what scripts are allowed to run
• Input validation with Joi — malicious strings stripped or rejected before reaching DB

MediumWhat is SQL injection? How does Prisma prevent it?⌄

SQL injection — attacker puts SQL code in a form field to manipulate the query. Classic example: email = "' OR 1=1 --"

Prisma uses parameterized queries internally. All values are passed as parameters, never concatenated into the SQL string. The DB treats them as data, never as SQL code.

// Prisma generates internally: // SELECT * FROM users WHERE email = ? (with value bound separately) // Attacker's SQL code is treated as a literal string, not executed
Deep DiveWhat is the AppError class and why did you create it?⌄

AppError extends the native Error class and adds two things: statusCode (HTTP status) and isOperational: true.

isOperational is the key flag. In the error handler, we check it:

• AppError (operational) — known business errors (user not found, invalid input). Send the message directly to the client.
• Unknown errors — bugs, DB crashes. Never expose internals. Send "Internal server error" and log the full stack trace.

This distinction prevents accidentally leaking stack traces, file paths, or DB query details in error responses.
Deep DiveWhy is the global error handler the LAST middleware in Express?⌄

Express identifies the global error handler by its 4-argument signature: (err, req, res, next). It only runs when next(err) is called from any previous middleware or route.

It must be registered after all routes because Express processes middleware in the order they're registered. If you put it before routes, it would never receive errors from routes that haven't been registered yet.

// app.js — order matters app.use(routes) // all routes first app.all('*', ...) // 404 catch app.use(errorHandler) // error handler LAST
Deep DiveWhat is the Prisma error handling in errorHandler.js? Why do you handle P2002 specially?⌄

Prisma throws typed errors with error codes. P2002 is a unique constraint violation — e.g. trying to register with an email that already exists.

Without special handling, Prisma would throw a raw DB error with table names and constraint names exposed. The error handler maps it to a human-readable message:

// P2002 constraint name contains "email" // → "A user with this email address already exists" // instead of raw Prisma error with DB internals

Other codes: P2025 (not found → 404), P2003 (FK violation → 400).

React & Redux 6 questions
BasicWhat is Redux Toolkit? What slices do you have?⌄

Redux Toolkit is the official, simplified way to use Redux for global state management. Instead of writing action types + action creators + reducers separately, you use createSlice.

In this project there are 3 slices:

• authSlice — current user, token, login/logout state
• attendanceSlice — today's record, check-in/out status
• leaveSlice — leave list, balance, pending requests

Why Redux and not just useState? These slices are needed across many components — the navbar, dashboard, attendance card all need auth info. Passing it via props everywhere (prop drilling) would be messy.
BasicWhat is a ProtectedRoute component? How does it work?⌄

ProtectedRoute is a wrapper component that checks if the user is authenticated before rendering a page. If not logged in, it redirects to the login page.

const ProtectedRoute = ({ children, roles }) => { const { user } = useAuth() if (!user) return <Navigate to="/login" /> if (roles && !roles.includes(user.role)) return <Navigate to="/403" /> return children }

It also checks the user's role — an employee trying to access the admin dashboard gets redirected to a 403 page.

MediumWhat is an Axios interceptor? How is it used here?⌄

An interceptor is a function that runs automatically before every request is sent or after every response is received — without repeating code in every API call.

Request interceptor — automatically adds the Authorization: Bearer <token> header to every API call. No need to manually add it everywhere.

Response interceptor — if a response comes back with 401 (token expired), automatically calls /auth/refresh to get a new token, then retries the original request. The user never sees the expiry.

MediumWhat does the useSocket hook do?⌄

The useSocket hook manages the Socket.io connection lifecycle:

• Connects to the server with the JWT access token on mount
• Registers event listeners (attendance:checkedIn, leave:approved, etc.)
• Disconnects and cleans up listeners when the component unmounts

Without cleanup, if the component re-renders, you'd register duplicate listeners and fire events multiple times — a common React bug with WebSockets.

MediumWhat does the useLiveTimer hook do?⌄

On mount, it calls GET /attendance/live-timer to get the check-in time and elapsedMs from the server. It then starts a setInterval that ticks every second to update the displayed time.

This is why the timer is accurate after page refresh — the starting offset comes from the server (which calculated it from the DB's checkInTime), not from a local variable that resets on refresh.

Deep DiveWhy do you have separate API modules (auth, attendance, leave) instead of one file?⌄

Separation of concerns — each module contains only the API calls related to one domain. Benefits:

• Easier to find: all leave API calls are in leave.js, not scattered across files
• Easier to change: if the base URL changes, you update one place per module
• Easier to test: you can mock the entire leave module in tests
• Readable imports: import { applyLeave } from '../api/leave' is self-documenting

Architecture & Patterns 6 questions
BasicWhat is MVC pattern? How is it applied in this project?⌄

MVC (Model-View-Controller) separates the application into 3 layers, each with one job:

• Model — data layer. In this project: Prisma schema + DB queries inside controllers.
• View — presentation layer. In this project: the React frontend.
• Controller — business logic. In this project: controllers/*.controller.js — receives request, queries DB, sends response.

Routes define which URL maps to which controller. Middlewares run before the controller.

BasicWhat is the consistent API response shape in this project?⌄

Every response — success or error — follows the same shape:

// Success { success: true, message: '...', data: { ... } } // Error { success: false, message: '...', errors: [...] }

This is done via sendResponse() and sendError() utility functions. The frontend always knows: check success first, then read data or message. No guessing the response structure.

MediumWhat is pagination? How did you implement it?⌄

Pagination splits large result sets into pages instead of returning all rows at once — prevents slow queries and large payloads.

// Query params: ?page=2&limit=20 const skip = (page - 1) * limit // skip first 20 rows prisma.attendance.findMany({ skip, take: limit }) prisma.attendance.count({ where }) // total for UI // Response includes pagination metadata { records, pagination: { total, page, limit, totalPages } }

Both queries run in Promise.all — parallel, not sequential.

MediumWhat is environment variables (.env)? What variables does this project use?⌄

Environment variables store configuration that changes between environments (dev/staging/prod) and secrets that should never be in source code.

In this project: DATABASE_URL, JWT_SECRET, JWT_REFRESH_SECRET, JWT_EXPIRES_IN, CORS_ORIGIN, PORT, NODE_ENV, rate limit values.

.env is in .gitignore — never committed. The .env.example file is committed to show teammates what variables they need to set, without exposing real values.
Deep DiveWhat is the leave overlap check? How does the query work?⌄

Before applying leave, we check if the employee already has a leave that overlaps the requested dates. Two date ranges overlap when:

// Range A: existing leave (startA → endA) // Range B: new request (startB → endB) // They overlap when: startA <= endB AND endA >= startB where: { status: { in: ['PENDING', 'APPROVED'] }, OR: [{ startDate: { lte: endB }, endDate: { gte: startB } }] }

This prevents an employee from applying for two leaves on overlapping dates.

Deep DiveWhy does leave approval skip weekends when creating attendance rows?⌄

When leave is approved for a date range (e.g. Monday–Friday), the server loops through every day and creates attendance rows. Saturday (day 6) and Sunday (day 0) are skipped:

if (day.getDay() === 0 || day.getDay() === 6) continue // 0 = Sunday, 6 = Saturday

Also, working days count (for balance deduction) excludes weekends. If you apply for Mon–Fri, it deducts 5 days, not 7. countWorkingDays() filters weekends before counting.

Project / HR Questions 6 questions
BasicTell me about this project. What does it do?⌄

WorkTrack is a workforce management system built with Node.js, Express, Prisma ORM, MySQL, and React. It manages three roles: Admin, Manager, and Employee.

Core features: employees mark daily attendance with GPS location capture and a live working-hours timer. They can apply for 6 types of leave — managers approve or reject their team's requests. Admins have a dashboard with real-time org-wide statistics.

Key technical aspects: JWT dual-token authentication, Socket.io for real-time events, RBAC with manager scope isolation, Prisma transactions for atomic operations, and 14+ security measures.

BasicWhat was the most challenging part of this project?⌄

Good answers to pick from (be specific, not vague):

1. Live timer across page refresh — realizing the timer state had to live in the DB, not in memory. Implementing the /live-timer endpoint that returns elapsed milliseconds.

2. Leave approval transaction — needing three separate DB operations to happen atomically. Learning about prisma.$transaction to ensure no partial state.

3. Manager scope isolation — understanding that role check alone isn't enough. A manager can't be allowed to see another manager's team data just because they share the same role.

MediumWhat would you improve if you had more time?⌄

Good answers (shows technical thinking):

1. Refresh token family / reuse detection — detect if a stolen refresh token is used (if old token is used, invalidate all sessions)
2. Automatic ABSENT marking — a cron job that marks employees absent if they didn't check in by end of day
3. Email notifications — email the employee when leave is approved/rejected, instead of only Socket event
4. Unit + integration tests — test each controller with a test database
5. Redis for rate limiting — current in-memory rate limiter resets on server restart. Redis persists across restarts.

MediumHow would you scale this if there were 10,000 employees?⌄

1. Database indexing — already done in this project (@@index on role, departmentId, managerId)
2. Redis caching — cache dashboard stats that are read frequently and change rarely
3. Socket.io with Redis adapter — if running multiple server instances, Socket rooms must be shared via Redis pub/sub
4. Connection pooling — Prisma already manages a connection pool, but configure pool size for load
5. Horizontal scaling — run multiple Node.js instances behind a load balancer (stateless design helps here — no session stored in memory)

MediumWhy did you choose MySQL over MongoDB for this project?⌄

This data is highly relational — users belong to departments, employees report to managers, attendance links to employees, leaves link to employees and reviewers, balances link to employees and years.

MySQL gives us: foreign key constraints (data integrity), JOINs for reports, UNIQUE constraints to prevent duplicate records, and transactions for atomic operations like leave approval.

MongoDB would be a poor fit here — you'd lose referential integrity, transactions are more complex, and the data structure is strongly typed with clear relationships.

Deep DiveHow does the attendance status get set when leave is approved?⌄

There's a mapping table in the leave controller:

const LEAVE_TO_ATTENDANCE = { CASUAL: 'LEAVE', SICK: 'LEAVE', HALF_DAY: 'HALF_DAY', EARLY_LEAVE: 'PRESENT', // still counts as present! WFH: 'WFH', FIELD_VISIT: 'FIELD_VISIT' }

EARLY_LEAVE maps to PRESENT because the employee came to work — they just left early. FIELD_VISIT and WFH don't deduct balance either — they're tracked in attendance but don't consume leave days.
