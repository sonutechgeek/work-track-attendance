API Testing Guide

41 endpoints across 6 modules. Follow the test flow below, or jump to any section.

Base URL
http://localhost:5000/api/v1
Auth Header
Authorization: Bearer {accessToken}
Content-Type
application/json
Health Check
GET http://localhost:5000/health
Admin admin@company.com Admin@1234
Manager manager@company.com Manager@1234
Employee employee@company.com Employee@1234

Recommended test order

  1. Login as Admin → copy accessToken from response
  2. Create a department → note the id
  3. Create a manager user → assign to department
  4. Create an employee user → assign to department + set managerId
  5. Login as Employee → get new accessToken
  6. Check-in (POST /attendance/check-in) → note the response
  7. Poll live-timer (GET /attendance/live-timer) → see elapsedMs grow
  8. Check-out (POST /attendance/check-out)
  9. Apply a leave (POST /leaves) → note leave id
  10. Login as Manager → approve the leave
  11. Login as Admin → check reports/dashboard

Auth

5 endpoints
POST /auth/login Get access + refresh tokens No Auth ▶
Rate-limited: 5 attempts per 15 minutes per IP.
Request Body
{
  "email": "admin@company.com",
  "password": "Admin@1234"
}
Response 200
{
  "success": true,
  "message": "Login successful",
  "data": {
    "accessToken": "eyJhbGciOiJIUzI...",
    "user": {
      "id": 1,
      "name": "Super Admin",
      "email": "admin@company.com",
      "role": "ADMIN"
    }
  }
}
Copy accessToken — set it as Authorization: Bearer {token} in all subsequent requests. The refreshToken is stored automatically as an httpOnly cookie.

POST /auth/refresh Rotate tokens — no body needed Cookie ▶
No body or Authorization header needed. The refreshToken cookie is sent automatically by the browser. In Postman, enable "Send cookies" and the cookie will be attached after login.
Response 200
{
  "success": true,
  "message": "Token refreshed",
  "data": { "accessToken": "eyJhbGciOiJIUzI..." }
}
PATCH /auth/me/password Change your password All roles ▶
Request Body
{
  "currentPassword": "Admin@1234",
  "newPassword": "Admin@5678"
}
Validation rules
newPassword requirements:
- min 8 characters
- at least 1 uppercase letter
- at least 1 lowercase letter
- at least 1 number
POST /auth/logout Invalidate session All roles ▶
Headers
Authorization: Bearer {accessToken}
No body needed. Clears the refresh token in DB and expires the cookie. The access token will expire naturally after 15 minutes.

POST /attendance/check-out Record check-out, compute working hours All roles ▶

GET /leaves/pending All pending leaves in your team AdminManager ▶
Manager sees only their team's pending leaves. Admin sees all pending leaves org-wide. Returns oldest-first so the most urgent reviews are at the top.
GET /leaves/:id Single leave record Scoped ▶
Employees can only fetch their own leaves. Managers can only fetch their team's leaves. Admins can fetch any leave.
POST /leaves/:id/approve Approve — atomic transaction AdminManager ▶
Request Body
{ "comments": "Approved. Enjoy." }
What happens inside the transaction
1. Leave status → APPROVED
2. Attendance rows created for
   each working day in the range
3. Leave balance *_used incremented
   (only for balance-deducting types)
All three DB operations succeed or all fail together (Prisma $transaction). Partial approvals are impossible.
Returns 403 if a Manager tries to approve a leave for someone not on their team.
POST /leaves/:id/reject Reject with mandatory reason AdminManager ▶
Request Body
{ "comments": "Insufficient leave balance for this period." }
comments is required when rejecting (min 3 chars). No balance is deducted on rejection.
DEL /leaves/:id Cancel own pending leave Employee ▶
Only works on PENDING leaves. Cannot cancel an already approved or rejected leave. Can only cancel your own leave.
