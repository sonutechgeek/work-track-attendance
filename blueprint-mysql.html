Defense in Depth

Security Architecture

JWT + Refresh Token Rotation
Access tokens expire in 15 minutes. Refresh tokens (7 days) live in an httpOnly; Secure; SameSite=Strict cookie — never in localStorage. On each refresh, the old token is revoked and a new one issued.
Password Hashing
bcryptjs with salt rounds of 12. Passwords are excluded from all Prisma selects via select: { password: false }. Password change requires providing the current password first.
SQL Injection Prevention via Prisma
Prisma uses fully parameterized queries — no raw string interpolation. User input never touches SQL directly. For the rare raw query (prisma.$queryRaw), Prisma's Prisma.sql template literal enforces parameterization.
Account Lockout
After 5 consecutive failed logins, the account is locked for 30 minutes via lock_until column. failed_logins counter resets on success. Admins can unlock accounts manually via PATCH.
Rate Limiting
Login endpoint: 5 requests / 15 min per IP. General API: 100 requests / 15 min per authenticated user. Remaining quota communicated via response headers. Uses express-rate-limit.
HTTP Security Headers
Helmet.js sets Content-Security-Policy, X-Frame-Options: DENY, X-Content-Type-Options: nosniff, Strict-Transport-Security, and removes X-Powered-By.
RBAC Middleware + Manager Scope
Every protected route chains authenticate (JWT verify) then authorize(...roles) (role check). Manager endpoints also run a scope guard: the Prisma query adds WHERE manager_id = req.user.id — a crafted request for another team's data returns 403.
Input Validation
Every POST/PATCH body is validated against a Joi schema before reaching the controller. Validation failures return a structured 400 with field-level messages. No unvalidated data reaches Prisma.
CORS Configuration
CORS is configured with an explicit whitelist of allowed origins (no wildcard in production). Credentials mode is enabled only for the frontend origin. Preflight responses are cached via max-age.
Audit Logging
All Create, Update, Delete operations are logged with Winston: actor userId, role, endpoint, payload shape (not values), and timestamp. Logs persist to rotating files for compliance review.

Engineering Decisions

Key Technical Decisions

Why Prisma over raw Sequelize for MySQL
Prisma's type-safe query builder means every field name and relation is checked at build time — a renamed column is a compile error, not a runtime crash. prisma migrate dev generates SQL migration files automatically from schema changes, giving a full version-controlled history of every ALTER TABLE. Sequelize requires more boilerplate for the same coverage. The trade-off: Prisma requires Node 16+ and adds a build step, but that's acceptable for this project.
Live Timer — Accurate Across Page Refreshes
The timer is not stored in frontend state alone. On check-in, the server saves check_in_time (DATETIME) to MySQL. On every page load, the employee dashboard calls GET /attendance/today. If a check-in exists with no check-out, the frontend computes elapsed = Date.now() - new Date(check_in_time) and starts the timer from that point. The displayed seconds update via a setInterval in a custom useTimer hook. No server polling needed.
Leave Approval — Prisma $transaction (ACID)
When a manager approves a leave, the server runs prisma.$transaction(async (tx) => { ... }): (1) update leaves.status = APPROVED, (2) create one attendances row per calendar day in the range with status: LEAVE, (3) update leave_balances to increment *_used. All three writes run inside a MySQL transaction. If any step throws, MySQL rolls back everything automatically — leave stays PENDING, no attendance rows are created, balance is unchanged.
Manager Scope — WHERE clause, not middleware filter
The frontend does not render out-of-scope links. The real enforcement is in the Prisma query: prisma.user.findMany({ where: { managerId: req.user.id } }). Even a raw API call with another employee's ID hits a scope guard middleware that queries prisma.user.findFirst({ where: { id: targetId, managerId: req.user.id } }) — if it returns null, the response is 403. The WHERE clause is the source of truth, not any UI gate.
Axios Token Refresh Interceptor
An Axios response interceptor catches 401 errors. It calls POST /auth/refresh (using the httpOnly cookie automatically), stores the new access token in memory (not localStorage), and retries the original request. This means the user is never prompted to log in again during a session — token refresh is invisible. Access tokens are stored in a module-level variable, not React state, to survive re-renders.
Geolocation — Browser API + Flat SQL Columns
The useGeolocation hook calls navigator.geolocation.getCurrentPosition() and wraps it in a Promise. Coordinates are sent with the check-in/check-out body as lat and lng. Unlike MongoDB's embedded object, MySQL stores these as separate DECIMAL columns (check_in_lat, check_in_lng) — this makes spatial filtering in SQL straightforward (e.g. WHERE check_in_lat BETWEEN ? AND ?). Leaflet.js renders the pin client-side.
Standard API Response Shape
Every endpoint returns a consistent envelope: {"success": true, "data": {...}, "message": "..."} on success and {"success": false, "error": "...", "details": [...]} on failure. The global errorHandler.js converts Prisma PrismaClientKnownRequestError (e.g. unique constraint P2002, FK violation P2003) into readable 400/409 responses. Frontend interceptors read response.data.data uniformly.
