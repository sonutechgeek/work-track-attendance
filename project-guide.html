01 — Overview

What is WorkTrack?

WorkTrack is a workforce management system built with Node.js + Express on the backend and React on the frontend. It manages three types of users — Admin, Manager, and Employee — each with different permissions and views.

Admin

Full access. Creates departments, manages all employees, views org-wide reports, approves any leave.

Manager

Scoped access. Sees only their assigned team. Approves/rejects their team's leaves.

Employee

Personal access. Marks attendance, applies for leave, sees own history and balance.

Core Features

FeatureWhat it does
AttendanceCheck-in/out with GPS location capture. Live timer synced via Socket.io.
Leave ManagementApply for 6 leave types. Manager/Admin approve/reject. Balance auto-deducted.
Real-time UpdatesSocket.io pushes events instantly — no page refresh needed.
RBACRole-based access + manager scope guard prevents data leaks between teams.
ReportsAdmin dashboard, monthly reports, leave balance summaries, attendance analytics.
SecurityJWT + refresh tokens, account lockout, rate limiting, Helmet, input validation.

05 — Database

Table Relationships Explained

RelationshipTypeDetail
users → departmentsMany-to-OneMany employees belong to one department. SET NULL on dept delete (employee record survives).
users → users (self)Self-referentialmanagerId points to another row in the same users table. One manager has many employees.
departments → users (head)One-to-OneA department can have one head employee. headId is a back-reference to users.

users → leavesOne-to-ManyOne employee can apply for many leaves. Also has a separate reviewedBy relation (who approved).
users → leave_balancesOne-to-ManyOne balance row per employee per year. UNIQUE(employeeId, year) prevents duplicates.

⚠

Important pattern: When a manager is deleted, all their employees' managerId becomes NULL (not deleted). When a department is deleted, users' departmentId becomes NULL. This is onDelete: SetNull — data is never accidentally lost.

06 — Authentication

Authentication System

Uses a dual-token strategy — a short-lived access token for API calls and a long-lived refresh token stored in an HttpOnly cookie.

Login Flow

1
POST /api/v1/auth/login

Email + password sent. Rate limited to 5 attempts per 15 min.

2
Account lock check

If lockUntil is in the future → reject with minutes remaining. HTTP 423.

3
bcrypt.compare(password, hash)

If wrong → increment failedLogins. At 5 failures → lock for 30 minutes.

4
Generate two tokens

Access token (15 min), Refresh token (7 days). Refresh token is bcrypt-hashed before saving to DB.

5
Response

Access token in JSON body. Refresh token in HttpOnly cookie (JS cannot read it).

Token Refresh Flow

// POST /api/v1/auth/refresh const token = req.cookies?.refreshToken // read from cookie jwt.verify(token, JWT_REFRESH_SECRET) // verify signature bcrypt.compare(token, user.refreshToken) // compare against stored hash // Issue new access token + rotate refresh token
ℹ

Why hash the refresh token? If your database leaks, the attacker gets only a bcrypt hash — they cannot use it to create new access tokens. The real token is only in the user's cookie.

Protected Route Flow

// Every protected route runs authenticate middleware: const token = req.headers.authorization.split(' ')[1] jwt.verify(token, JWT_SECRET) // verify + decode prisma.user.findUnique({ where: { id } }) // confirm user still exists if (!user.isActive) → 403 // check account active req.user = user // attach to request
07 — Access Control

RBAC — Role-Based Access Control

Two layers of access control protect every endpoint. Role guard checks WHO you are. Scope guard checks WHO you can access.

ADMIN
  • All employees across org
  • All attendance records
  • All leave requests
  • Create departments/users
  • Admin dashboard + reports
  • Manual attendance correction
MANAGER
  • Only their team (managerId check)
  • Team attendance records
  • Team leave requests
  • Approve/reject team leaves
  • Team reports
EMPLOYEE
  • Own profile only
  • Own attendance history
  • Apply / cancel own leave
  • View own leave balance
  • Live timer (check-in/out)

How the code works

// Layer 1 — Role Guard // Usage: router.get('/users', authenticate, authorize('ADMIN'), ...) const authorize = (...roles) => (req, res, next) => { if (!roles.includes(req.user.role)) → 403 next() } // Layer 2 — Manager Scope Guard // Checks that target employee belongs to this manager const target = await prisma.user.findFirst({ where: { id: targetId, managerId: req.user.id } }) if (!target) → 403 // "Employee not in your team"
✓

Admins bypass scope guard automatically. The scope guard has if (req.user.role === 'ADMIN') return next() at the top — admins can access any employee without a managerId check.

08 — Attendance

Attendance System

Employees check in and out once per day. The system captures GPS location at both events and calculates working hours automatically.

Check-In Flow

1
Browser captures GPS

Frontend uses navigator.geolocation to get lat/lng before the API call.

2
POST /api/v1/attendance/check-in

Body: { lat, lng, address }. Server validates. Checks if already checked in today.

3
Prisma upsert

Creates or updates the attendance row for today. UNIQUE(employeeId, date) ensures one row per day.

4
Socket.io event emitted

emitToUser(employeeId, 'attendance:checkedIn', { checkInTime }) — live timer starts instantly on screen.

Live Timer — How it survives page refresh

// GET /api/v1/attendance/live-timer // Called when employee's page loads const record = await prisma.attendance.findUnique(...) return { active: true, checkInTime: record.checkInTime, elapsedMs: Date.now() - new Date(record.checkInTime).getTime() // ↑ Frontend resumes timer from this exact ms offset }

Check-Out Calculation

const workingHours = parseFloat( ((checkOutTime - existing.checkInTime) / 3600000).toFixed(2) ) // 3600000 = milliseconds in one hour // e.g. 9 hours 30 min = 9.50 working hours

Leave Application Flow

1
Overlap check

Checks if employee already has a PENDING or APPROVED leave that overlaps the requested date range.

2
Balance check

Counts working days (excludes weekends). Checks if total - used >= requested days.

3
Leave created

Saved with status PENDING. Socket event sent to manager + admin rooms.

4
Manager/Admin approves

Three things happen inside a single prisma.$transaction — atomically.

Approval Transaction — 3 atomic operations

await prisma.$transaction(async (tx) => { // 1. Update leave status → APPROVED await tx.leave.update({ data: { status: 'APPROVED', reviewedById, reviewedAt } }) // 2. Create attendance row for each day in the leave range for (const day of leaveDays) { if (isWeekend(day)) continue // skip Saturday/Sunday await tx.attendance.upsert({ data: { status: 'LEAVE', note: 'CASUAL leave approved' } }) } // 3. Deduct balance (only for types that deduct) await tx.leaveBalance.updateMany({ data: { casualUsed: { increment: workingDays } } }) })
ℹ

Why use $transaction? All three DB operations either succeed together or all fail together. If step 3 fails, steps 1 and 2 are automatically rolled back. This prevents leave being marked approved but balance not deducted.

10 — Reports

Reports & Admin Dashboard

All report queries run in parallel using Promise.all for performance.

EndpointWhoReturns
GET /reports/dashboardAdminTotal employees, present today, on leave, pending leaves, monthly attendance breakdown by status
GET /reports/attendance-summaryAdmin/ManagerPer-employee count of each status (PRESENT/ABSENT/LEAVE) for a date range
GET /reports/leave-summaryAdmin/ManagerLeave balance remaining for each employee (total vs used per type) for a given year
GET /reports/monthlyAdminAttendance breakdown by status + leave breakdown by type for any month/year

Dashboard query uses 6 parallel DB calls

const [totalEmployees, totalDepts, presentToday, pendingLeaves, leavesToday, monthlySummary] = await Promise.all([ prisma.user.count(...), prisma.department.count(...), prisma.attendance.count(...), // present today prisma.leave.count(...), // pending approvals prisma.attendance.count(...), // on leave today prisma.attendance.groupBy(...) // monthly status breakdown ])

Promise.all runs all 6 queries at the same time. Dashboard loads as fast as the slowest single query, not the sum of all 6.

11 — Real-time

Socket.io — Real-time Events

How the connection is established

1
Frontend connects after login

io('http://localhost:5000', { auth: { token: accessToken } })

2
Server JWT middleware fires

Verifies the token. Attaches socket.userId and socket.userRole.

3
User joins their private room

socket.join('user:5'). Admins also join the 'admin' broadcast room.

4
Connection stays open

One persistent connection for the entire session. Server can push events any time.
